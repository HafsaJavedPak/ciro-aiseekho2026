CIRO: Complete Hackathon Execution Plan
1. Deep Challenge Analysis
Core Technical Challenges
The real difficulty here isn't any single component — it's the integration surface. Each piece (signal fusion, classification, allocation, simulation, notification) is straightforward in isolation. The challenge is making them feel like a coherent, flowing intelligence rather than five disconnected modules bolted together.
The second hidden difficulty is Google Antigravity. Judges will look for genuine orchestration — agents that actually reason, delegate, and hand off — not a single LLM call you've labeled "multi-agent." This distinction will separate top teams from the rest.
The third challenge is demoability under pressure. Live demos at hackathons fail constantly. Your architecture must be able to run on cached/mock data without breaking the illusion.
What Most Teams Will Build
* A single backend LLM call that classifies crisis from combined text
* A map with markers
* Hardcoded "resource allocation" that's just a sorted list
* Antigravity used as a label, not real orchestration
* A web dashboard, forgetting the mobile-first mandate
* Beautiful UI with fragile data pipelines
How You Differentiate
* Real Antigravity traces shown live in the app — the agent reasoning process visible to judges
* Two simultaneous crises with resource conflict resolution happening in real-time, visually
* A false alarm recovery arc in your demo that shows the system self-correcting — this alone impresses judges because no one builds it
* Explainable decisions — every allocation decision shows why, not just what
* The mobile app as command center, not afterthought

Data Flow Explanation
Signal → Decision → Action → Feedback is the core loop:
1. Signals arrive from 3 sources every 30–60 seconds (or on-demand trigger for demo)
2. Ingestion service normalizes format, deduplicates, and scores credibility
3. Normalized events are pushed to a queue (Redis or in-memory for MVP)
4. Antigravity orchestrator is triggered; Signal Fusion Agent aggregates into incident context
5. Classification Agent determines crisis type, severity, confidence
6. If confidence > threshold: Severity Agent + Allocation Agent run in parallel
7. Simulation Agent computes before/after state for each planned action
8. Messaging Agent generates stakeholder-specific notifications
9. All decisions, traces, and state changes persist to Firestore
10. Mobile app receives real-time updates via WebSocket
11. If contradictory signals arrive → Recovery Agent re-evaluates and may retract
Event Lifecycle
RAW_SIGNAL → NORMALIZED → CREDIBILITY_SCORED → FUSED 
→ CLASSIFIED → SEVERITY_ASSESSED → RESOURCES_ALLOCATED 
→ ACTIONS_SIMULATED → NOTIFICATIONS_SENT → MONITORING 
→ (UPDATE or CLOSE or RETRACT)

4. Google Antigravity Integration Strategy
Important note: Google Antigravity is Google's multi-agent orchestration framework (the ADK — Agent Development Kit). Use the Python ADK with Gemini as the underlying model. All agents are defined as ADK Agent objects with explicit tools, system instructions, and handoff logic.
Agent Design
Principle: Each agent has ONE job, ONE clear input/output contract, and explicit tool access. Avoid agents that do everything.
# Agent registry - define once, reference everywhere


AGENTS = {
    "signal_fusion":       SignalFusionAgent,
    "credibility":         CredibilityAgent,
    "classification":      ClassificationAgent,
    "severity_prediction": SeverityAgent,
    "resource_allocation": ResourceAgent,
    "simulation":          SimulationAgent,
    "stakeholder_msg":     StakeholderAgent,
    "recovery":            RecoveryAgent,
}


Agent Responsibilities
Signal Fusion Agent
* Input: List of raw normalized signals from last N minutes
* Job: Group signals by location cluster, identify co-occurring signals, compute signal coherence score
* Output: IncidentContext object (location, signal_types, combined_confidence, raw_signals[])
* When: Runs first, every cycle
* AI vs Rule: Geospatial clustering is rule-based; coherence assessment is LLM-driven
Credibility & Conflict Agent
* Input: IncidentContext
* Job: Score each signal source (known source weight, language urgency, geolocation precision, velocity of mentions). Detect contradictions (e.g., social posts say flood, field report says burst pipe)
* Output: CredibilityReport (per-signal scores, contradiction flags, recommended verification steps)
* When: Runs after fusion, before classification
* AI vs Rule: Urgency language scoring and contradiction detection are LLM-driven; source weights are rule-based lookup table
Crisis Classification Agent
* Input: IncidentContext + CredibilityReport
* Job: Classify crisis type, assign severity level (1–5), compute overall confidence (0–1), extract affected area
* Output: CrisisClassification (type, severity, confidence, location, affected_radius_km, affected_population)
* When: Core classification step
* AI vs Rule: Type classification is LLM with structured output; severity thresholds are rule-augmented
Severity & Prediction Agent
* Input: CrisisClassification + historical context
* Job: Predict evolution (peak time, spread risk, duration estimate, uncertainty range). Identify vulnerable populations in radius
* Output: SeverityPrediction (peak_time, duration_hours, spread_risk, uncertainty_pct, vulnerable_population)
* When: Runs in parallel with Resource Allocation
* AI vs Rule: Peak time/duration uses heuristic formulas + LLM adjustment; spread risk is LLM reasoning
Resource Allocation Agent
* Input: CrisisClassification + all active crises + current resource state
* Job: Assign resources to incidents, resolve conflicts when two crises compete for same unit, explain tradeoffs
* Output: AllocationPlan (assignments[], conflicts_detected[], tradeoff_explanations[])
* When: Runs after classification; re-runs if new crisis emerges
* AI vs Rule: Conflict detection and tradeoff explanation are LLM-driven; scoring formula is deterministic
Simulation Agent
* Input: AllocationPlan + SeverityPrediction
* Job: For each planned action, compute before state → after state, estimate response time delta, congestion impact, side effects
* Output: SimulationResult[] (action, before, after, response_time_improvement, congestion_delta, side_effects[])
* When: Runs after allocation
* AI vs Rule: Mostly rule-based with parameterized formulas; LLM used only for side-effect narrative
Stakeholder Messaging Agent
* Input: CrisisClassification + AllocationPlan + SimulationResult
* Job: Generate tailored messages for each stakeholder type
* Output: StakeholderMessages {public_alert, emergency_services_brief, hospital_notification, utility_alert, media_release, command_summary}
* When: Runs after simulation
* AI vs Rule: Pure LLM with structured prompts per audience type
Recovery / Correction Agent
* Input: New incoming signals + existing CrisisClassification
* Job: Detect if new evidence contradicts existing classification, trigger re-classification, generate retraction if needed
* Output: CorrectionAction (action_type: CONFIRM|UPDATE|RETRACT, updated_classification, retraction_message)
* When: Monitors continuously; triggered on contradictory signals
* AI vs Rule: Contradiction detection is LLM; retraction threshold is rule-based
Agent Communication Flow
Orchestrator receives trigger
  → SignalFusionAgent.run(signals)
      → CredibilityAgent.run(incident_context)
          → ClassificationAgent.run(context, credibility_report)
              ┌──────────────────────────────────┐
              │ Parallel execution                │
              ├── SeverityAgent.run(classification)
              └── ResourceAgent.run(classification, active_crises, resources)
              └──────────────────────────────────┘
          → SimulationAgent.run(allocation_plan, severity_prediction)
          → StakeholderAgent.run(classification, allocation, simulation)
  → Persist all traces to Firestore
  → Push update to WebSocket clients


Continuously:
  → RecoveryAgent.monitor(new_signals, active_incidents)
      → If contradiction detected: re-trigger ClassificationAgent

credibility_score = (

velocity_score = min(mentions_last_5min / 20, 1.0)

6. Crisis Detection & Prediction Logic
Classification Pipeline
Step 1: Spatial Clustering (Rule-based)
  → Group signals within 1.5km radius
  → Minimum 2 signals to form candidate incident


Step 2: Type Voting (Hybrid)
  → Count crisis_type_hints per cluster
  → Weight by credibility
  → Leading type becomes primary hypothesis
  → LLM confirms or overrides with reasoning if votes are close


Step 3: Confidence Calculation (Formula)
  confidence = (
      avg_credibility_of_signals    * 0.40 +
      signal_count_score            * 0.25 +
      source_diversity_score        * 0.20 +
      (1 - contradiction_factor)    * 0.15
  )
  signal_count_score = min(signal_count / 5, 1.0)
  source_diversity_score = unique_source_types / total_source_types


Step 4: Severity Scoring (Rule + LLM)
  severity_level (1-5) based on:
  - confidence threshold
  - crisis type base severity
  - sensor breach magnitude
  - affected area size
  - time of day (peak hours = higher severity)


Step 5: LLM Classification Confirmation
  Prompt: "Given these signals, classify this crisis. Output JSON only."
  Model output: {type, severity, confidence, reasoning, counter_hypothesis}


Severity Scoring
def compute_severity(crisis_type, confidence, signal_data) -> int:
    base_severity = {
        "urban_flooding": 3,
        "heatwave": 3,
        "road_accident": 2,
        "power_outage": 2,
        "fire": 4,
        "infrastructure_failure": 3,
    }
    
    score = base_severity.get(crisis_type, 2)
    
    # Modifiers
    if confidence > 0.85: score += 1
    if signal_data.sensor_breach_magnitude > 1.5: score += 1
    if signal_data.mention_velocity > 20: score += 1
    if signal_data.time_of_day in ["peak_morning", "peak_evening"]: score += 1
    
    return min(score, 5)

def predict_evolution(crisis_type, severity, location):
    params = CRISIS_EVOLUTION_PARAMS[crisis_type]
    affected_radius = 0.5 + (severity * 0.3) + random.gauss(0, 0.1)
    peak_time = datetime.now() + timedelta(hours=params["peak_time_hours"])
    duration = params["duration_hours"] * (0.8 + severity * 0.1)
    spread_radius_at_peak = affected_radius + (params["spread_rate_km_per_hour"] * params["peak_time_hours"])
    affected_population = estimate_population(location, spread_radius_at_peak)
    
    return SeverityPrediction(
        peak_time=peak_time,
        duration_hours=duration,
        affected_radius_km=affected_radius,
        spread_radius_km=spread_radius_at_peak,
        affected_population=affected_population,
        uncertainty_pct=max(10, 30 - (confidence * 20)),
    )

Prioritization Algorithm
Use a weighted priority score — no linear programming, no genetic algorithms. Fast, explainable, demo-friendly:
def compute_incident_priority(incident):
    return (
        incident.severity              * 0.35 +
        incident.confidence            * 0.20 +
        incident.affected_population_k * 0.25 +  # in thousands
        incident.vulnerability_index   * 0.10 +  # 0-1 (low-income, elderly population)
        incident.spread_risk           * 0.10
    )


def allocate_resources(incidents, resources):
    # Sort incidents by priority (descending)
    sorted_incidents = sorted(incidents, key=compute_incident_priority, reverse=True)
    
    allocation_plan = []
    for incident in sorted_incidents:
        required = CRISIS_RESOURCE_REQUIREMENTS[incident.crisis_type][incident.severity]
        assigned = {}
        conflicts = []
        
        for resource_type, quantity_needed in required.items():
            available = resources[resource_type]["available"]
            if available >= quantity_needed:
                assigned[resource_type] = quantity_needed
                resources[resource_type]["available"] -= quantity_needed
            elif available > 0:
                assigned[resource_type] = available
                conflicts.append({
                    "resource": resource_type,
                    "needed": quantity_needed,
                    "assigned": available,
                    "deficit": quantity_needed - available,
                    "reason": f"Already committed to {[i.id for i in sorted_incidents if i != incident]}"
                })
                resources[resource_type]["available"] = 0
            else:
                conflicts.append({"resource": resource_type, "needed": quantity_needed, "assigned": 0})
        
        allocation_plan.append(AllocationResult(incident_id=incident.id, assigned=assigned, conflicts=conflicts))
    
    return allocation_plan


Travel Time Estimation
def estimate_travel_time(resource_location, incident_location, crisis_type):
    distance_km = haversine(resource_location, incident_location)
    # Base speed 40 km/h urban, adjusted for congestion
    congestion_factor = 1.3 if crisis_type in ["urban_flooding", "road_accident"] else 1.0
    travel_time_min = (distance_km / 40) * 60 * congestion_factor
    return round(travel_time_min, 1)


Multi-Crisis Conflict Resolution
When two crises need the same ambulance:
1. Both incidents scored
2. Higher-priority incident gets full allocation first
3. Deficit for lower-priority incident is logged as a conflict
4. LLM generates the tradeoff explanation for the trace: "Rescue Team Alpha assigned to G-10 flooding (severity 4, pop. 8,400) rather than I-9 accident (severity 3, pop. 200) due to higher combined priority score. ETA to I-9 via Police Unit Bravo as substitute: 7 minutes."
5. Escalation flag set if deficit exceeds 50% of required resources

    before_state: dict
    action_taken: str
    after_state: dict

11. AI/LLM Strategy
Model Choice
Use Gemini 1.5 Flash as primary — it supports structured JSON output, has a generous free tier, and integrates natively with ADK.
Keep Gemini 1.5 Pro on standby for the most complex reasoning tasks (contradiction detection, recovery agent) — but only call it when needed to preserve rate limits.
Structured Output Strategy
Never ask the LLM to produce free-form text for anything that feeds into business logic. Use Pydantic schemas enforced via function calling / response schema:
from google.generativeai import GenerativeModel

        "severity": {"type": "integer", "minimum": 1, "maximum": 5},
        "confidence": {"type": "number", "minimum": 0, "maximum": 1},
        "reasoning": {"type": "string"},
        "counter_hypothesis": {"type": "string"},

Prompt Engineering Pattern
Use a consistent structure for all agent prompts:
SYSTEM: You are [Agent Name], a specialized component of CIRO (Crisis Intelligence & Response Orchestrator). Your ONLY job is [specific task]. Output ONLY valid JSON matching the provided schema.


CONTEXT:
- City: [city]
- Active incidents: [count]
- Current resources: [summary]


INPUT DATA:
[structured signal/incident data]


TASK:
[specific instruction]


CONSTRAINTS:
- Do not hallucinate location names
- If confidence < 0.4, flag for verification
- Always include reasoning field
- Output JSON only, no markdown


Reliability Techniques
* Temperature = 0 for all classification and allocation agents (deterministic)
* Temperature = 0.3 for stakeholder messaging (some creativity allowed)
* Schema enforcement via response_schema parameter — rejects non-conforming output
* Retry with exponential backoff on any API failure (3 retries max)
* Fallback response if all retries fail: use rule-based output, log as "degraded mode"
* Validation layer after every LLM response: if schema validation fails, use fallback
Cost Optimization
For a hackathon demo with ~20 agent runs, you're well within free tier. But defensively:
* Cache identical signal combinations (hash input → cache output for 5 minutes)
* Use Flash for 80% of calls, Pro only for contradiction resolution
* Limit stakeholder message generation to once per crisis (don't regenerate on every signal update)
* Compress signal context: summarize signals before sending (rule-based summarizer, not LLM)

14. Robustness & Failure Handling
API Downtime
class ResilientSignalSource:
    def __init__(self, primary, fallback_cache_ttl=300):
        self.primary = primary
        self.cache = {}
        self.last_success = {}
    
    async def fetch(self, params):
        try:
            result = await self.primary.fetch(params)
            self.cache[params] = (result, time.time())
            return result
        except Exception as e:
            if params in self.cache:
                cached, ts = self.cache[params]
                if time.time() - ts < self.cache_ttl:
                    return CachedResult(cached, stale=True)
            return DegradedResult(error=str(e))


Conflicting Signals
Already handled in the Credibility & Conflict Agent. Key principle: never silently pick one — always log the conflict and show it to the operator.
Duplicate Incidents
def is_duplicate(new_incident, existing_incidents, distance_threshold_km=1.0, time_window_min=30):
    for existing in existing_incidents:
        if (haversine(new_incident.location, existing.location) < distance_threshold_km and
            abs((new_incident.timestamp - existing.timestamp).total_seconds()) < time_window_min * 60 and
            new_incident.crisis_type == existing.crisis_type):
            return True, existing.id
    return False, None
